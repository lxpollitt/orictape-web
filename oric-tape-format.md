# Oric cassette tape format

A reference for the **fast-format** cassette encoding produced by the Oric-1
BASIC **1.0** ROM and Oric Atmos BASIC **1.1b** ROM, as reproduced by
orictape-web's "save as audio" feature.

The details here have been read directly off the ROMs, verified on a
cycle-accurate 6502 emulator, and/or confirmed by phase and bit-exact 
round-tripping against genuine Oric tape recordings. The exception is
the analogue interface (§6 section) which was drawn from publicly
available schematics.

Companion documents: [oric-asm-syntax.md](oric-asm-syntax.md),
[oric-asm-rom-v10-v11b-reference.md](oric-asm-rom-v10-v11b-reference.md). The
encoder is in [src/audioEncoder.ts](src/audioEncoder.ts).

> **ROM scope.** This describes the **1.0** (Oric-1) ROM. The Atmos **1.1b** ROM
> uses byte-for-byte the same tape-write *logic* - same framing, same 3 explicit
> stop bits, same timer mechanism - at relocated addresses; the only behavioural
> difference found is the tape-speed flag (zero-page `$67` in 1.0 vs `$024D` in
> 1.1b). Cycle-count constants such as `G0` (§5) are 1.0-specific.

---

## 1. Saving: the VIA timer and bit cells

The cassette waveform is generated by the 6522 VIA's **Timer 1** running freely
and toggling output line **PB7**. Each time T1 underflows it reloads from its
latch, toggles PB7, and sets its interrupt flag. The CPU programs the T1 latch
for each half-cycle and polls the flag to stay in step. Because the VIA is 
clocked by the CPU's phi2, the timer and the instruction stream never drift 
relative to one another - so the whole format is deterministic to the cycle.

Two latch values are used:

| half-cycle  | latch         | period (CPU cycles)    |
|-------------|---------------|------------------------|
| short       | `$00D0` = 208 | 210 (latch + 2 reload) |
| long        | `$01A0` = 416 | 418                    |

A **bit cell** is two half-cycles. In the context of the CSAVE algorithm (see
later notes on CLOAD differences) the leading half-cycle is *always* short, 
and the trailing half-cycle effectively encodes the bit:

| bit | cell          | meaning                                                 |
|-----|---------------|---------------------------------------------------------|
| `1` | short + short | full 2400 Hz cycle                                      |
| `0` | short + long  | 2400 Hz half-cycle + 1200 Hz half-cycle (1600 Hz mean)  |

A `0` is *not* a symmetric 1600 Hz cycle - "1600 Hz" is just the harmonic mean of
its two halves.

In a **48 kHz** audio file, the half-periods are a clean **10 samples (short)** 
and **20 samples (long)** - so a `1`-bit is 20 samples and a `0`-bit is 30. 
(10/20 samples is 208/417 us, i.e. the latch values; the ~2-cycle reload overhead 
is dropped, which is far inside the decoder's tolerance.) 48 kHz is our preferred 
audio file sample rate because every half-period lands on an integer sample.

From the chip's point of view this is a square wave. By the time it is recorded
it is not - see §6.

When `CSAVE` is invoked, due to the way the code initialises the VIA and first
synchronises with the timer, it produces a single `1`-bit output (low short + high 
short) immediately before the first bit of the first byte of the program save is 
generated. This is always a `0`-bit (low short + high long) - the start bit of the
first sync byte.

## 2. Byte framing

Each byte is sent **least-significant-bit first**, framed as:

```
[ start = 0 ][ d0 d1 d2 d3 d4 d5 d6 d7 ][ parity ][ stop = 1, 1, 1 ]
```

- **Start bit** is a `0` cell.
- **8 data bits**, LSB first.
- **Parity** = `NOT(popcount(data) & 1)` - a byte with an *even* number of `1`
  data bits gets parity `1`. The ROM computes it with `EOR #$01` / `LSR A`.
- **Stop bits**: at least 3 stop bits. (Normally alternating between 3 and 4 
  stop bits per byte with a short-long/long-short phase shift for all the 
  `0`-bits in the byte, or 3.5 stop bits per byte with a polarity phase change across
  all the bits in the byte, depending on how you look at it - they are the same thing.
  The "gap" byte is the single exception to this rule. See §4 and §5.)

Example - the sync byte `$16` = `%00010110`: data LSB-first = `0 1 1 0 1 0 0 0`,
popcount 3 (odd) so parity `0`, giving the frame `0 / 01101000 / 0 / 111`.

`PutTapeByte` is at `$E5C6` in the 1.0 ROM.

## 3. Block structure

A saved program is a single block:

```
[ 0x16 x N (sync bytes) ][ 0x24 sync marker ][ header fixed size fields ][ header program name, 0x00-terminated ][ program data ][ extra byte ]
```

- **Leader / sync** A run of 259 `0x16` bytes then one `0x24` marker. 
- **Header**
  - **Fixed size fields** 9 bytes:
    `[ reserved1, reserved2, fileType, autorun, end_hi, end_lo, start_hi, start_lo, reserved3 ]`,
    addresses big-endian. The program data length is implicit (`end - start`).
    `fileType` is one of: `0x00` (BASIC); `0x40` (array - only supported by Oric
    Atmos / 1.1 ROM via the `STORE`/`RECALL` commands); `0x80` (machine code / 
    memory block). For BASIC and machine code / memory block files, the `reserved`
    bytes are unused (set from uninitialised RAM locations on save, and ignored on 
    load). For array files, `reserved1` and `reserved2` specify the array length.
    An `autorun` byte value of anything other than zero is interpreted as autorun 
    on. On the save side, the value used for autorun varies by ROM. The 1.0 ROM 
    uses `0x04` (the option bytes scan index in the 1.0 code) and the 1.1 ROM uses 
    `0xC7` (the keyword token for `AUTO`).
  - **Name**, the file name in Oric ASCII, NUL-terminated.
- **Program data**, then the **extra byte**: the ROM's data-write loop (end-check
  at `$E554`) compares the running pointer with the end address and loops *while
  pointer < end*, so it emits `start..end` **inclusive**. The header's end
  address is *exclusive* (so one past the `00 00` program terminator in the case
  of a BASIC program), so the loop writes one byte *at* that address - one past 
  the terminator. On real hardware its value is whatever was in RAM there; 
  orictape-web emits a fixed `$52` for determinism so two subsequent saves of the 
  same file produce the same result.

There is no checksum and no end-of-file marker - the loader reads exactly
`end - start` bytes.

## 4. The inter-byte half-cycle: 3/4 cadence and phase

### Saving
The LDY #$04 loop at the end of PutTapeByte makes four WriteBit calls - the first
emits the parity bit, the other three are forced to 1 by SEC.

However, the T1 timer runs continuously, continuing to emit idle fast half-cycles,
while the CPU runs its inter-byte code before the next `PutTapeByte` re-syncs
to the timer; this results in at least one extra half-cycle per byte. This
equates to an extra single half-cycle per byte (so 3.5) for all bytes, except
for the "gap" byte (which executes substantially more inter-byte code, resulting 
in a variable number of additional half-cycles in that one case - see §5).

The start bit, data bits, and parity bit, are all then generated by the Oric's
save algorithm using short-short (for `1`-bits) or short-long (for `0`-bits) as
described in §1. The extra half cycle that is generated per byte means the phase 
polarity of these bits alternates between each byte (switching between high first 
half-cycles vs low first half-cycles for all bits within the byte).

### Loading

In contrast, on the loading side, the code uses a fixed polarity bit cell model, 
where each bit always starts with a high half-cycle followed by a low half-cycle.

The effect of this is that it alternates (per byte) between seeing
- 3 stop bits vs 4 stop bits (rather than it seeing the 3.5 stop bits per byte)
- short-long vs long-short phase shift for all the `0`-bits in the byte (rather 
  than the phase polarity changing per byte, and `0`-bits always being short-long).

While the saving and loading structural paradigms are different, they are 100%
equivalent in real terms and the result is bitwise identical. 

We use the loading paradigm for simplicity in the remainder of this document.
(And orictape-web uses this same paradigm throughout, including both loading
and saving.)

### Predictability

Because the 6522 VIA is clocked by the CPU's phi2, the ROM's load and save 
algorithms are 100% deterministic. And because the save algorithm always output
a fixed number of sync preamble ($16) bytes followed by one sync marker ($24),
the sync marker byte always preceded by 3 stop bits, and its `0`-bits are always
short-long phase. (Assuming the recording/playback chain does not invert the 
phase polarity - see note in §6 for more details.)

orictape-web reproduces this predictability by toggling 3/4 stop bits per byte 
cadence and choosing the start phase - from the parity of its hard-coded 
sync-byte count - so the `0x24` before the header lands after 3 stop bits. This 
cadence is adjusted as a special case at the "gap" byte (see §5).

## 5. The name->data gap

The one place the plain 3/4 stop bits toggle does not follow the simple 
altenerating per byte cadence is at boundary between program **name** (which
is the last field in the header) and the program **data**. At this point
in addition to the standard inter-byte code, the saving algorithm executes
additional code to: clear the status line, print "Saving " (including
associated paper and ink code) and **print the program name to the screen**,
and set up the data pointer. The T1 timer continues to drive tone transitions
while this code is running, before the code returns to its normal byte writing
loop, re-syncing with the timer, to write out the program data bytes.

The name print is the only part that varies with the program: the screen-output
routine (`$F82F`) pokes each character in a loop costing exactly **19 CPU cycles
per character**. So the gap is:

```
gap_cycles(nameLen) = G0 + 19 * nameLen
```

with **G0 = 648** fixed cycles (emulator-verified). In execution order:

| chunk         | cycles       | what the ROM does                                                                  |
|---------------|--------------|------------------------------------------------------------------------------------|
| fixed chunk 1 | 590          | finish the name terminator ($00) byte's write, clear the status line,              |
|               |              | print "Saving ", set up the name print                                             |
| variable      | 19 x nameLen | poke each program-name character to the screen                                     |
| fixed chunk 2 | 58           | finish the name print, set up the data pointer, start the write of first data byte |

(`G0 = 590 + 58 = 648`. In reality there are a further few-cycles of irreducible 
cost *not* included in this calculation: the latency between the timer firing and
the CPU's `BVC` noticing it - 0-6 cycles, phase-dependent. It only nudges `G0` 
within `[631, 687]`, and every value in that window gives the identical result 
for any realistic name length, so it has no effect.)

The first data byte re-syncs to the next timer timeout, so the whole gap reduces
to one integer `q`:

```
q      = floor(gap_cycles / 210)                       # idle short half-periods in the gap
e      = nameLen & 1
prec   = 3 + e                                         # byte before the gap - just the normal toggle
first  = (q + 2) - (1 if (q even and e == 1) else 0)   # stop run before the first data byte (this is what fills the "gap")
second = (3 + e) if (q even) else (4 - e)              # stop run before the next byte; q odd flips the post-gap phase
```

`q`'s magnitude sets the first data byte's preceding stop bits run base (`q + 2`) 
and its parity flips the post-gap 3/4 stop bits cadence phase. After the first
data byte the plain 3/4 stop bits toggle the second data byte is preceded by
`second` stop bits, with the 3/4 stop bits toggling as normal per byte after
that.

The above was validated to reproduce the stop-bits and phase behaviour
of a wide selection of measured real-tape program name lengths.

## 6. The analogue interface 

### A note on phase

The above discussion on phase assumes that the recording and playback chains
have identical phase polarity. If the phase polarity differs between the two 
then the result of the decode is identical in terms of bits and bytes.
But phase related aspects all switch around: what was a high half-cycle becomes 
a low half-cycle; `0`-bits that were read as long-short become short-long, and 
vice-versa; and the 3/4 stop bit cadence reverses such that bytes that would 
have been read as having 3 stop bits are now read as having 4, and vice-versa. 

### Why the recorded signal on tape is not a square wave

> Component values are from the Oric service-manual schematic / a community KiCad
> redraw, not measured first-hand. The R12/R13 *divider* (below) is corroborated
> by the manual's own wording ("attenuated by R12 and R13"); the LM358 input by
> its stated ~50 mV hysteresis.

On PB7 the signal is nominally a hard square wave. Three things stand between it
and a decoded program: the Oric output stage, the cassette tape channel, and the 
Oric input stage.

**Output (save) stage - a mild low-pass.** `PB7 -> R12 (22K, series) -> tape-out`,
with **R13 (1K) and C7 (47nF) both to ground** at the tape-out node: R12/R13 form
a divider (setting the ~150 mV output level the manual quotes), C7 is the shunt
capacitor. It's a one-pole RC low-pass whose corner is `1/(2π·R·C7)` with the
resistance C7 sees `R = R12 ∥ R13 ∥ Z_load`:
- Unloaded, `R = R12 ∥ R13 ≈ 957 Ω -> 3.5 kHz` - this is the corner *floor*.
- The recorder's mic input `Z_load` sits in *parallel*, so loading *raises* the
  corner: a ~600 Ω–1 kΩ mic input gives ~7–9 kHz.

Either way the corner is well above the 2400 Hz carrier. The output stage only
slightly **softens the square wave corners**. It does *not* produce the deep,
rounded "U" style shapes that can be found on some original Oric tape-recordings.

**The cassette tape channel - NOT modelled.** The additional smoothing and shaping
in a real recording is greatly influenced by recording process (recording device 
and settings, and the tape type). Measured across several different tapes *and* 
two different digitisers (so it's in the signal, not the capture): (a) a steep 
band-limit at ~3.5 kHz - the carrier's 3rd harmonic (7200 Hz) is 30–50 dB down,
so there are no sharp edges left on tape; (b) a high-pass "droop" (fast attack,
slow release) common to magnetic recording, where the head EMF is proportional 
to `dΦ/dt` (it differentiates, +6 dB/oct - the reason tape needs playback EQ);
(c) an occasional attack-side overshoot from device resonance. The net is roughly
a band-pass that keeps the carrier and discards the higher harmonics, giving smooth
quasi-sinusoidal humps. A least-squares fit of a one-pole low-pass, or a low-pass
+ high-pass band-pass, to a real sample cycle does *not* reproduce the shape; 
faithful reproduction would ideally use real tape-channel model, which is deemed
out of scope for orictape-web currently.

**Input (load) stage.** `tape-in -> C6 (47nF, AC-coupling) -> LM358 op-amp
comparator (~50 mV hysteresis) -> the VIA's read input`. It strips DC,
amplifies/clips, and recovers bits from edge/zero-crossing **timing with
hysteresis** - insensitive to waveform shape. So a clean square, a softened
square, or the tape-coloured U all load, provided the half-period edge timings
are preserved. (Hence, for playback *into* a real Oric, the output shape barely
matters - only the edge timings do.)

For playback into a real Oric from a PC line-out, use a moderate level (well
below full scale) - the input is single-rail with no over-voltage protection.

## 7. What orictape-web emits

48 kHz, 16-bit, mono PCM, implementing all of the above:

- bit cells at the 10/20-sample half-periods of §1, then the whole stream run
  through the §6 output-stage one-pole low-pass (~6.9 kHz mic-load corner,
  normalised to ±16384 = -6 dBFS, ~6 dB of headroom) - a mild corner-softening,
  the authentic Oric output; the tape colouring is deliberately not modelled;
- the §2 framing, with the `NOT(popcount & 1)` parity;
- a 259-byte `0x16` leader + one `0x24`;
- the §4 3/4 cadence, with the §5 name->data gap formula;
- **the §4 phase**: a `0`-bit's long half-cycle is placed low or high to match a
  real save - **high iff the byte's preceding stop-bit run is even**, except the
  first data byte, which takes the *opposite* of the (normal) 2nd data byte (its 
  own run is the anomalous gap value);
- the `$52` extra byte (§3), then a trailing `0` and ~50 ms of trailing
  silence so the receiver latches the final bit of the extra byte;
- header handling identical to a TAP save (autorun byte normalised, end address
  auto-corrected), so a WAV and a TAP of the same program carry identical bytes.

It is validated by a **bit and phase-exact round-trip** test: decode a genuine
recording, re-encode it, decode that, and confirm the bit streams match the original
bit-for-bit (bar the documented header normalisation) including checking the phase 
of each bit.
