# Oric cassette tape format

A reference for the **fast-format** cassette encoding produced by the Oric-1
BASIC **1.0** ROM and Oric Atmos BASIC **1.1b**, as reproduced by
orictape-web's "save as audio" feature.

Every factual claim here has been read directly off the ROM, verified on a
cycle-accurate 6502 emulator, and/or confirmed by bit-exact round-tripping
against genuine tape recordings. The one section drawn from secondary sources
(the analogue interface, §7) is flagged as such.

Companion documents: [oric-asm-syntax.md](oric-asm-syntax.md),
[oric-asm-rom-v10-v11b-reference.md](oric-asm-rom-v10-v11b-reference.md). The
encoder is in [src/audioEncoder.ts](src/audioEncoder.ts).

> **ROM scope.** This describes the **1.0** (Oric-1) ROM. The Atmos **1.1b** ROM
> uses byte-for-byte the same tape-write *logic* - same framing, same 3 explicit
> stop bits, same timer mechanism - at relocated addresses; the only behavioural
> difference found is the tape-speed flag (zero-page `$67` in 1.0 vs `$024D` in
> 1.1b). Cycle-count constants such as `G0` (§5) are 1.0-specific.

---

## 1. The signal: the VIA timer and bit cells

The cassette waveform is generated by the 6522 VIA's **Timer 1** running freely
and toggling output line **PB7**. Each time T1 underflows it reloads from its
latch, toggles PB7, and sets its interrupt flag; the CPU programs the latch for
each half-period and polls the flag to stay in step. Because the VIA is clocked
by the CPU's phi2, the timer and the instruction stream never drift relative to
one another - the whole format is deterministic to the cycle.

Two latch values are used:

| half-period | latch         | period (CPU cycles)    |
|-------------|---------------|------------------------|
| short       | `$00D0` = 208 | 210 (latch + 2 reload) |
| long        | `$01A0` = 416 | 418                    |

A **bit cell** is two half-periods. The leading half is *always* short; the
trailing half encodes the bit:

| bit | cell          | meaning                                              |
|-----|---------------|-----------------------------------------------------|
| `1` | short + short | one full ~2400 Hz cycle                             |
| `0` | short + long  | an asymmetric short-then-long cell (~1600 Hz mean)  |

A `0` is *not* a symmetric 1600 Hz cycle - "1600 Hz" is just the harmonic mean of
its two halves.

**At 48 kHz**, the half-periods are a clean **10 samples (short)** and **20
samples (long)** - so a `1`-bit is 20 samples and a `0`-bit is 30. (10/20 samples
is 208/417 us, i.e. the latch values; the ~2-cycle reload overhead is dropped,
which is far inside the decoder's tolerance.) 48 kHz is chosen because every
half-period lands on an integer sample.

From the chip's point of view this is a square wave. By the time it is recorded
it is not - see §7.

## 2. Byte framing

Each byte is sent **least-significant-bit first**, framed as:

```
[ start = 0 ][ d0 d1 d2 d3 d4 d5 d6 d7 ][ parity ][ stop = 1, 1, 1 ]
```

- **Start bit** is a `0` cell.
- **8 data bits**, LSB first.
- **Parity** = `NOT(popcount(data) & 1)` - a byte with an *even* number of `1`
  data bits gets parity `1`. The ROM computes it with `EOR #$01` / `LSR A`; the
  textbook "even/odd parity" label is misleading here, so implement the formula,
  not the label.
- **Stop bits**: the ROM emits exactly **3**. (The `LDY #$04` loop at the end of
  `PutTapeByte` makes four `WriteBit` calls - the first emits the parity bit, the
  other three are forced to `1` by `SEC`.) The number of `1`-cells *actually seen*
  before the next start bit is not always 3 - see §4.

Example - the sync byte `$16` = `%00010110`: data LSB-first = `0 1 1 0 1 0 0 0`,
popcount 3 (odd) so parity `0`, giving the frame `0 / 01101000 / 0 / 111`.

`PutTapeByte` is at `$E5C6` in the 1.0 ROM.

## 3. Block structure

A saved program is a single block:

```
[ 0x16 x N (leader) ][ 0x24 ][ 9-byte header ][ name, 0x00-terminated ][ program data ][ extra byte ]
```

- **Leader / sync.** A run of `0x16` bytes then one `0x24` marker. The 1.0 ROM
  emits **6** `0x16` bytes; a loader needs only ~4 clean ones to lock on.
  (orictape-web emits 256 for a healthy analogue lock-in margin.)
- **Header**, 9 bytes:
  `[ 0x00, 0x00, fileType, autorun, end_hi, end_lo, start_hi, start_lo, 0x00 ]`,
  addresses big-endian. The block length is implicit (`end - start`).
- **Name**, ASCII, NUL-terminated.
- **Program data**, then the **extra byte**: the ROM's data-write loop (end-check
  at `$E554`) compares the running pointer with the end address and loops *while
  pointer < end*, so it emits `start..end` **inclusive**. The header's end
  address is *exclusive* (one past the `00 00` program terminator), so the loop
  writes one byte *at* that address - one past the terminator. On real hardware
  its value is whatever was in RAM there; orictape-web emits a fixed `$52` for
  determinism.

There is no checksum and no end-of-file marker - the loader reads exactly
`end - start` bytes.

## 4. The inter-byte half-cycle: 3/4 cadence and phase

Although the ROM writes only 3 explicit stop bits, the number of `1`-cells the
loader sees between bytes alternates **3, 4, 3, 4 ...** all the way along the
stream. That, and the long-half phase in §7, are the same single fact: **the
inter-byte gap is an odd number of half-cycles.**

It comes to effectively **3.5 "sync bits" - 7 half-cycles**: the 3 stop bits the
ROM writes, plus one half-cycle left over. (The timer free-runs continuously,
emitting idle `1`-cells, while the CPU runs a fixed inter-byte loop before the
next `PutTapeByte` re-syncs; that loop is not a whole number of cells, so half a
cycle is always left over.)

This has **nothing to do with absolute polarity.** The save side is polarity-
agnostic, and the loader - exactly like our own decoder - reads each cycle by its
*length*, ignoring sign. What the leftover half-cycle does is shift the **phase**
by half a cycle every byte, which the loader sees two ways:

- the inter-byte gap rounds to **3 or 4** whole `1`-cells, alternately - the 3/4
  cadence; and
- a `0`-bit's long half-cycle falls **low or high**, alternately from one byte to
  the next (§7).

Same half-cycle, seen twice: once as a length, once as an alignment. (It also
explains why the run count is *length*-dependent but *content*-independent -
identical-length names give identical cadence.)

The alternation is **anchored: the run immediately before the `0x24` marker is
always 3** (confirmed across many genuine saves). orictape-web reproduces it by
toggling 3/4 per byte and choosing the start phase - from the parity of its
hard-coded sync-byte count - so the `0x24` lands on 3.

## 5. The name->data gap

The one place the plain 3/4 toggle is perturbed is the boundary between the
program **name** and the program **data**. There the ROM does a burst of
non-tape work - finish the name byte's write, clear the status line, print
"Saving ", **print the program name to the screen**, set up the data pointer -
all while the timer free-runs idle tone, before the first data byte re-syncs.

The name print is the only part that varies with the program: the screen-output
routine (`$F82F`) pokes each character in a loop costing exactly **19 CPU cycles
per character**. So the gap is:

```
gap_cycles(nameLen) = G0 + 19 * nameLen
```

with **G0 = 648** fixed cycles (emulator-verified). In execution order:

| chunk         | cycles       | what the ROM does                                                                       |
|---------------|--------------|-----------------------------------------------------------------------------------------|
| fixed chunk 1 | 590          | finish the name byte's write, clear the status line, print "Saving ", set up the name print |
| variable      | 19 x nameLen | poke each program-name character to the screen                                          |
| fixed chunk 2 | 58           | finish the name print, set up the data pointer, write the first data byte               |

(`G0 = 590 + 58 = 648`. A further few-cycle, irreducible cost is *not* included:
the latency between the timer firing and the CPU's `BVC` noticing it - 0-6 cycles,
phase-dependent. It only nudges `G0` within `[631, 687]`, and every value in that
window gives the identical result for any realistic name length, so it has no
effect.)

The first data byte re-syncs to the next timer timeout, so the whole gap reduces
to one integer `q`:

```
q      = floor(gap_cycles / 210)                       # idle short half-periods in the gap
e      = nameLen & 1
prec   = 3 + e                                         # byte before the gap - just the normal toggle
first  = (q + 2) - (1 if (q even and e == 1) else 0)   # stop run on the first data byte
second = (3 + e) if (q even) else (4 - e)              # next byte; q odd flips the post-gap phase
```

`q`'s magnitude sets the first data byte's stop-run base (`q + 2`); its parity
flips the post-gap cadence phase. After the first data byte the plain 3/4 toggle
resumes from `second`. This closed form reproduces every measured real-tape name
length exactly; `q` steps up roughly every 11 name characters (the first step
falls between nameLen 10 and 11, which is why short names flip and long names
stay in phase).

## 6. The analogue interface (why the recorded signal is not a square wave)

> Component values are from the Oric service-manual schematic / a community KiCad
> redraw, not measured first-hand. The R12/R13 *divider* (below) is corroborated
> by the manual's own wording ("attenuated by R12 and R13"); the LM358 input by
> its stated ~50 mV hysteresis.

On PB7 the signal is a hard square wave. Three things stand between it and a
decoded program: the Oric output stage, the cassette tape channel, and the Oric
input stage. The surprise (measured below) is that the **output stage barely
shapes anything** - almost all the smoothing in a real recording is the tape.

**Output (save) stage - a mild low-pass.** `PB7 -> R12 (22K, series) -> tape-out`,
with **R13 (1K) and C7 (47nF) both to ground** at the tape-out node: R12/R13 form
a divider (setting the ~150 mV output level the manual quotes), C7 is the shunt
capacitor. It's a one-pole RC low-pass whose corner is `1/(2π·R·C7)` with the
resistance C7 sees `R = R12 ∥ R13 ∥ Z_load`:
- Unloaded, `R = R12 ∥ R13 ≈ 957 Ω -> 3.5 kHz` - this is the corner *floor*.
- The recorder's mic input `Z_load` sits in *parallel*, so loading *raises* the
  corner: a ~600 Ω–1 kΩ mic input gives ~7–9 kHz.

Either way the corner is well above the 2400 Hz carrier, so the output stage only
**softens the cell corners by a sample or two**. It does *not* produce the deep,
rounded "U".

**The cassette tape channel - NOT modelled.** The dramatic smoothing in a real
recording is the record/playback process, not the Oric. Measured across several
different tapes *and* two different digitisers (so it's in the signal, not the
capture): (a) a steep band-limit at ~3.5 kHz - the carrier's 3rd harmonic
(7200 Hz) is 30–50 dB down, so there are no sharp edges left on tape; (b) a
high-pass "droop" (fast attack, slow release) intrinsic to magnetic playback,
where the head EMF is proportional to `dΦ/dt` (it differentiates, +6 dB/oct - the
reason tape needs playback EQ); (c) an occasional attack-side overshoot from
playback resonance. The net is roughly a band-pass that keeps the carrier and
discards the harmonics, giving smooth quasi-sinusoidal humps. A least-squares fit
of a one-pole low-pass - or a low-pass + high-pass band-pass - to a real cell does
*not* reproduce it; faithful reproduction needs a real tape-channel model, which
is a rabbit hole and out of scope.

**Input (load) stage.** `tape-in -> C6 (47nF, AC-coupling) -> LM358 op-amp
comparator (~50 mV hysteresis) -> the VIA's read input`. It strips DC,
amplifies/clips, and recovers bits from edge/zero-crossing **timing with
hysteresis** - insensitive to waveform shape. So a clean square, a softened
square, or the tape-coloured U all load, provided the half-period edge timings
are preserved. (Hence, for playback *into* a real Oric, the output shape barely
matters - only the edge timings do.)

For playback into a real Oric from a PC line-out, use a moderate level (well
below full scale) - the input is single-rail with no over-voltage protection.

No difference was found between the Oric-1 and Atmos analogue stages.

## 7. What orictape-web emits

48 kHz, 16-bit, mono PCM, implementing all of the above:

- bit cells at the 10/20-sample half-periods of §1, then the whole stream run
  through the §6 output-stage one-pole low-pass (~6.9 kHz mic-load corner,
  normalised to ±16384 = -6 dBFS, ~6 dB of headroom) - a mild corner-softening,
  the authentic Oric output; the tape colouring is deliberately not modelled;
- the §2 framing, with the `NOT(popcount & 1)` parity;
- a 256-byte `0x16` leader + one `0x24`;
- the §4 3/4 cadence, with the §5 name->data gap formula;
- **the §4 phase**: a `0`-bit's long half-cycle is placed low or high to match a
  real save - **high iff the byte's sync run is even**, except the first data
  byte, which takes the *opposite* of the (normal) 2nd data byte (its own run is
  the anomalous gap value). We render "long-half-high" by emitting the long half
  first; since a cycle is read by total length, that is invisible to the loader
  and to our decoder (both read by length, never sign) - it only makes the
  waveform match what a real Oric wrote;
- the `$52` extra byte (§3), then a terminating `0` and ~50 ms of trailing
  silence so the receiver latches the final bit;
- header handling identical to a TAP save (autorun byte normalised, end address
  auto-corrected), so a WAV and a TAP of the same program carry identical bytes.

It is validated by a **bit-exact round-trip** test: decode a genuine recording,
re-encode it, decode that, and confirm the bit streams match the original
byte-for-byte (bar the documented header normalisation) - cadence, framing,
parity, and the name->data gap all included.  It also checks each byte's
**phase** (which half of the 0-cell is the long one) against the recording -
the byte values are phase-blind, so this is what proves the re-encode stays in
phase with what a real Oric wrote.
